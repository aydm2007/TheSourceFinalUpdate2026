# دليل قفل الـ AST Mutex

## الهدف
يهدف هذا الدليل إلى توثيق كيفية طلب وإصدار وإلغاء القفل (Mutex) على ملفات المصدر داخل مشروع **TheSource** لتجنب تعارضات تحرير الوكلاء المتزامنة.

## المصطلحات
- **AST Mutex**: قفل على شجرة التحليل التجريدي (Abstract Syntax Tree) لملف محدد.
- **Agent ID**: معرف الوكيل الذي يطلب القفل.
- **Lock Range**: نطاق الأسطر المتأثرة بالقفل داخل الملف.

## خطوات طلب القفل
1. **تحديد الملف والنطاق**
   ```json
   {
     "file_path": "<مسار_الملف>",
     "line_range": [<سطر_البداية>, <سطر_النهاية>]
   }
   ```
2. **استدعاء أداة `AstMutexLockManager`** مع المعطيات السابقة:
   ```
   AstMutexLockManager({
     agent_id: "<معرف_الوكيل>",
     file_path: "src/example.js",
     line_range: [10, 50]
   })
   ```
3. **التحقق من نجاح العملية** عبر فحص سجل `shadow_ledger.jsonl` أو استجابة الأداة التي يجب أن تكون `status: success`.

## خطوات إلغاء القفل
1. استدعاء نفس الأداة مع `action: "release"` (الإجراء الافتراضي هو طلب القفل، لذلك لإلغاء القفل نمرر `release: true`).
2. مثال:
   ```
   AstMutexLockManager({
     agent_id: "<معرف_الوكيل>",
     file_path: "src/example.js",
     line_range: [10, 50],
     release: true
   })
   ```
3. تأكيد الإلغاء عبر سجل `shadow_ledger.jsonl`.

## ملاحظات هامة
- لا يجوز طلب قفل لنفس النطاق من أكثر من وكيل في نفس الوقت؛ سيُرفض الطلب ويُسجل في `shadow_ledger.jsonl` كتحذير.
- يجب إلغاء القفل فور الانتهاء من التعديل لتجنب حجب الوكلاء الآخرين.
- في حالة فشل الوكيل أو انقطاع الجلسة، يُمكن للـ **Swarm Auto‑Healer** اكتشاف القفل العالق وإصداره تلقائياً بعد مهلة محددة (30 ثانية).

## دمج القفل في سير العمل
1. **EnterPlanMode** → وضع خطة التعديل.
2. طلب القفل باستخدام `AstMutexLockManager`.
3. تنفيذ التعديل (مثلاً `FileEdit` أو `SurgicalDiff`).
4. **ExitPlanMode** → إغلاق الخطة.
5. إلغاء القفل.
6. توثيق العملية في `shadow_ledger.jsonl`.
