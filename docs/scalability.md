# Scalability Guidelines

## Overview
This document outlines strategies to ensure the chat dashboard scales efficiently under increasing load.

## 1. Front‑end Performance
- **Code Splitting**: Use dynamic `import()` for heavy modules (e.g., audio processing).
- **Lazy Load Assets**: Load audio files and large images only when needed.
- **Virtualize Chat List**: Render only visible chat bubbles using a virtual list library.
- **Cache API Responses**: Store recent Sigma API replies in `sessionStorage` with a TTL of 5 minutes.

## 2. Backend / API Layer
- **Rate Limiting**: Apply a token‑bucket limiter (e.g., 30 requests/min per IP) to the Sigma proxy.
- **Batch Requests**: Group multiple user messages into a single request when possible.
- **Horizontal Scaling**: Deploy the API behind a load balancer (NGINX/HAProxy) with at least 2 instances.
- **Stateless Design**: Keep chat state client‑side; server only processes the message payload.

## 3. Infrastructure
- **Containerization**: Dockerize the front‑end server and Sigma proxy for easy scaling.
- **Auto‑Scaling Rules**: On Kubernetes, set CPU utilization threshold at 70 % to spin up new pods.
- **CDN for Static Assets**: Serve `dashboard.html`, CSS, JS, and audio files via a CDN (e.g., Cloudflare).

## 4. Monitoring & Observability
- **Metrics**: Expose Prometheus metrics for request latency, error rate, and active connections.
- **Alerting**: Trigger alerts when 95th‑percentile latency exceeds 300 ms or error rate > 1 %.
- **Logging**: Include request IDs in logs to trace end‑to‑end flows.

## 5. Load Testing
- **Tool**: Use `k6` or `artillery` for synthetic traffic.
- **Scenario**: Simulate 500 concurrent users sending a message every 2 seconds.
- **Success Criteria**:
  - Average response time < 200 ms.
  - No more than 2 % failed requests.

## 6. Future Enhancements
- **WebSocket Chat**: Switch from HTTP polling to WebSocket for real‑time bi‑directional communication.
- **Edge Computing**: Deploy inference models closer to users to reduce latency.
